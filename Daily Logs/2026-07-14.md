## Work Completed

### Pi-hole Deployment (Network-Wide Ad Blocking)

Goal: stand up Pi-hole as a lightweight DNS-based ad blocker for the entire home network, using an LXC container rather than a full VM since Pi-hole doesn't need its own kernel and benefits from fast boot/low overhead as always-on DNS infrastructure.

* Chose deployment method: Proxmox Community Scripts one-liner install over building the LXC manually (see [Proxmox Community Scripts over Manual LXC Creation for Pi-hole](../Decisions/Proxmox%20Community%20Scripts%20over%20Manual%20LXC%20Creation%20for%20Pi-hole.md))
* Planned container specs: LXC (not VM), static IP `192.168.86.202/24`, gateway `192.168.86.1`, IPv6 disabled (`none`), 512MB RAM / 1 core / 2GB disk, FUSE and TUN/TAP both disabled, hostname `pihole`

#### Troubleshooting

Hit a run of unrelated tooling issues getting the install to actually complete — full detail on each is in the Troubleshooting folder rather than here:

* Pasting the install command into the Proxmox noVNC console produced corrupted input instead of running it — [Bracketed-Paste Corruption in Proxmox noVNC Console](../Troubleshooting/Bracketed-Paste%20Corruption%20in%20Proxmox%20noVNC%20Console.md)
* Keypresses stopped registering entirely at a couple of points, in both the noVNC console and a PowerShell SSH session — turned out to be accidental text selection freezing input (Windows QuickEdit Mode) — [QuickEdit Mode and noVNC Selection Freezing Console Input](../Troubleshooting/QuickEdit%20Mode%20and%20noVNC%20Selection%20Freezing%20Console%20Input.md)
* Recurring `curl: Could not resolve host: raw.githubusercontent.com` failures mid-install — confirmed as a known, transient issue with these community scripts generally, not a local config problem — [raw.githubusercontent.com Intermittent DNS Resolution Failures](../Troubleshooting/raw.githubusercontent.com%20Intermittent%20DNS%20Resolution%20Failures.md)
* Piping the install script straight into `bash` silently swallowed an interactive yes/no confirmation prompt, auto-aborting the container each time — [Piped Install Script Swallowing Interactive Confirmation Prompt](../Troubleshooting/Piped%20Install%20Script%20Swallowing%20Interactive%20Confirmation%20Prompt.md)
* `pct exec` couldn't find the `pihole` binary by name even though it existed — a `PATH` issue, not a missing install — [pct exec Not Resolving Binary via PATH](../Troubleshooting/pct%20exec%20Not%20Resolving%20Binary%20via%20PATH.md)
* Declined the installer's optional Unbound add-on for now — [Unbound Deferred as a Separate Follow-Up Project](../Decisions/Unbound%20Deferred%20as%20a%20Separate%20Follow-Up%20Project.md)

### Result

* Pi-hole fully installed and confirmed working end-to-end: `pihole status` responds, admin dashboard reachable and logged into successfully at `http://192.168.86.202/admin`
* Blocklist populated automatically during install (StevenBlack/hosts list, ~76,000 domains)
* Admin password set via `pihole setpassword`

### Network-Wide DNS — Blocked on Access

* Home network uses a Spectrum modem paired with **Google Wifi** as the actual router/mesh system (not Spectrum's own router)
* The Google Wifi network is set up under his brother's Google account; brother is currently out of town
* Confirmed (via research, not yet executed) that Google Wifi/Google Home network management is fully cloud-based — the brother does **not** need to be physically home or on the same Wi-Fi to grant access
* Planned path: brother opens the Google Home app → profile icon → Add Home Member → invites by email → grants **Admin** access (Member may be permission-gated for network settings). Heads-up for the brother: Google's permission model isn't granular, so Admin access grants full control of the whole Home, not just networking.
* Once access is granted: Google Home app → Wifi network → Network settings → Advanced Networking → DNS → Custom → Primary DNS `192.168.86.202`. Secondary DNS decision deferred — leaving it blank means the whole network loses internet if Pi-hole goes down; setting a public fallback (e.g. `1.1.1.1`) avoids that but silently bypasses ad-blocking/logging during any Pi-hole downtime. Leaning toward a temporary public fallback for now given this is a new, unproven setup.
