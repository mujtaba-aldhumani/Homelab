## Work Completed

### Pi-hole Deployment (Network-Wide Ad Blocking)

Goal: stand up Pi-hole as a lightweight DNS-based ad blocker for the entire home network, using an LXC container rather than a full VM since Pi-hole doesn't need its own kernel and benefits from fast boot/low overhead as always-on DNS infrastructure.

* Chose deployment method: Proxmox Community Scripts (VE Helper-Scripts) one-liner install over building the LXC manually — reasoned that manual `pct create` container-creation practice was already being built via the Windows/Ubuntu VMs, so the marginal learning value of doing it manually again for Pi-hole was low; opted to spend the time on Pi-hole/DNS concepts instead
* Planned container specs: LXC (not VM), static IP `192.168.86.202/24`, gateway `192.168.86.1`, IPv6 disabled (`none`), 512MB RAM / 1 core / 2GB disk, FUSE and TUN/TAP both disabled (no need for either — no virtual filesystem mounting or VPN tunneling inside this container), hostname `pihole`

#### Troubleshooting — a long one

* **Bracketed-paste terminal glitch:** pasting the install command into the Proxmox noVNC web console repeatedly produced literal `^[[200~...~` escape codes instead of running the command. Root cause: bracketed-paste mode signaling not being stripped properly by that console. Worked around by typing commands manually or piping via `wget ... | bash` instead of `bash -c "$(wget ...)"`.
* **QuickEdit Mode freezing input:** several times mid-wizard, keypresses (Enter, arrow keys) stopped registering entirely in both the browser noVNC console and a Windows PowerShell SSH session. Root cause identified via forum/first-principles troubleshooting: accidental text selection (a stray click-drag) freezes a Windows console window's input entirely until the selection is cleared — this is QuickEdit Mode, enabled by default in PowerShell/cmd. Fix: click to clear the selection, and disable QuickEdit Mode going forward via right-click title bar → Properties → Options tab → uncheck QuickEdit Mode.
* **Wrong subnet on one build attempt:** one container build attempt showed `192.168.82.202/24` instead of `192.168.86.202/24` in the wizard summary — inconsistent with the saved `pihole.vars` defaults file, which was confirmed correct (`192.168.86.202`, gateway `192.168.86.1`) both before and after. Concluded this was a one-off wizard display glitch rather than a persistent bad config, since later runs correctly showed `.86.202`.
* **`curl: (6) Could not resolve host: raw.githubusercontent.com`:** hit this recurring error multiple times, at the point where the community script (and later, Pi-hole's own installer) tries to fetch content from GitHub's raw content CDN right after container network comes up. Ruled out a host-level DNS problem (Proxmox host itself resolved and pinged the same hostname fine). Researched via Proxmox forums/GitHub issues afterward and confirmed this is a widely-documented, known-flaky failure mode for these community scripts generally (not Pi-hole-specific, not specific to this homelab) — commonly linked to IPv6 route flakiness immediately after a fresh container's network comes up, or GitHub's raw-content CDN itself being intermittently unstable. No permanent fix applied; subsequent runs succeeded cleanly (all "Git DNS" checks passed) without further changes, supporting the "transient" theory. Documented fix if it recurs: disable IPv6 at the OS level inside the container (`net.ipv6.conf.all.disable_ipv6 = 1` via sysctl) or force IPv4-only curl calls.
* **Piped script swallowing an interactive confirmation prompt:** running the install via `wget -qLO - <url> | bash` caused a `[y/N]` confirmation inside the script (Pi-hole's own "external installer" disclaimer) to be silently answered "no" every time, auto-aborting the container. Root cause: piping into `bash` consumes stdin, so there's nothing left for an interactive prompt inside the script to read from your actual keyboard. Fix: download the script to a file first (`wget -qL <url> -O pihole.sh`), then run it as a separate command (`bash pihole.sh`) so stdin stays connected to the terminal.
* **`pct exec` couldn't find the `pihole` binary despite it existing:** `pct exec 102 -- which pihole` and `pct exec 102 -- pihole setpassword` both failed with "No such file or directory," even though the file was confirmed present and executable at `/usr/local/bin/pihole`. Root cause: `pct exec` doesn't always load the full container `PATH`. Fix: call the binary by its full path (`pct exec 102 -- /usr/local/bin/pihole setpassword`), which worked immediately.
* Declined the Pi-hole installer's optional prompt to add **Unbound** (a recursive DNS resolver that would let Pi-hole resolve directly from root DNS servers instead of forwarding to an upstream provider like Google/Cloudflare) — reasoned this is a distinct, valuable follow-up topic (privacy/no third-party DNS visibility) better tackled as its own deliberate project once the core setup is proven stable, rather than bundled into tonight's install.

### Network-Wide DNS — Blocked on Access

* Home network uses a Spectrum modem paired with **Google Wifi** as the actual router/mesh system (not Spectrum's own router)
* The Google Wifi network is set up under his brother's Google account; brother is currently out of town
* Confirmed (via research, not yet executed) that Google Wifi/Google Home network management is fully cloud-based — the brother does **not** need to be physically home or on the same Wi-Fi to grant access
* Planned path: brother opens the Google Home app → profile icon → Add Home Member → invites by email → grants **Admin** access (not Member — Admin is needed for DNS/network settings; Member may be permission-gated). Noted as a heads-up for the brother: Google's permission model isn't granular, so Admin access grants full control of the whole Home (all members/devices), not just networking.
* Once access is granted: Google Home app → Wifi network → Network settings → Advanced Networking → DNS → Custom → Primary DNS `192.168.86.202`. Secondary DNS decision deferred — leaving it blank means the whole network loses internet if Pi-hole goes down; setting a public fallback (e.g. `1.1.1.1`) avoids that but silently bypasses ad-blocking/logging during any Pi-hole downtime. Leaning toward a temporary public fallback for now given this is a new, unproven setup.
