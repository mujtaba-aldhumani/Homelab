### Dante SOCKS5 Proxy Setup

Goal: run a dedicated SOCKS5 proxy on the Ubuntu VM so a Stremio + Real-Debrid setup could (eventually) be shared without exposing a full Tailscale exit node.

* Installed Dante via `sudo apt install dante-server -y` — clean install, no errors
* Config plan: listen only on the Tailscale interface (`tailscale0`, port 1080), send traffic out via the regular LAN interface (`ens18`), require username/password auth, and allow any device in Tailscale's CGNAT range (`100.64.0.0/10`)
* **Mistake/troubleshooting:** attempted to edit `/etc/danted.conf` with `nano`, planning to clear the default sample/comment content first. Ran into confusion over which nano shortcuts actually select/delete all text (several suggested shortcuts weren't valid nano commands), and the terminal session became unresponsive to paste attempts. Closed the SSH session rather than keep fighting it.
* **Solution:** reconnected via a fresh SSH session and bypassed the interactive editor entirely, writing the full config in one shot with a heredoc:
  ```
  sudo tee /etc/danted.conf > /dev/null << 'EOF'
  logoutput: syslog

  internal: tailscale0 port = 1080
  external: ens18

  socksmethod: username

  client pass {
      from: 100.64.0.0/10 to: 0.0.0.0/0
  }

  socks pass {
      from: 100.64.0.0/10 to: 0.0.0.0/0
  }
  EOF
  ```
  Verified the written file with `cat /etc/danted.conf` before proceeding. This heredoc technique was reused later in the session for the sysctl persistence fix as well.
* Created a dedicated, no-shell proxy account for SOCKS5 auth rather than reusing a personal login:
  ```
  sudo useradd -r -s /usr/sbin/nologin proxyuser
  sudo passwd proxyuser
  ```
  * **Mistake:** first password-set attempt failed ("Sorry, passwords do not match") due to a mistyped confirmation entry; second attempt succeeded.
* Enabled and started the service: `sudo systemctl enable --now danted` — `systemctl status danted` confirmed `active (running)` with worker processes spawned correctly and no errors.
* Verified the proxy end-to-end from the personal laptop using curl through the SOCKS5 proxy:
  ```
  curl.exe -x socks5://proxyuser:<password>@100.74.175.81:1080 https://ifconfig.me
  ```
  * **Mistake:** first attempt used plain `curl` inside PowerShell, which is aliased to `Invoke-WebRequest` and doesn't understand the `-x` proxy flag, throwing a parameter-binding error. Corrected by explicitly calling `curl.exe` to invoke the real curl binary shipped with Windows rather than the PowerShell alias.
  * Result: request succeeded and returned the home network's public IP, confirming traffic was actually routing out through the VM.

### Stremio Proxy Integration — Corrected Approach

* Initial plan was to point Stremio's own settings directly at the Dante SOCKS5 proxy.
* **Mistake:** assumed Stremio had a proxy-configuration field under its Streaming settings. The "URL" and "Streaming HTTPS endpoint" options found there actually configure the connection to Stremio's own local streaming server (the torrent/debrid playback engine), and are unrelated to routing traffic through an external proxy.
* Researched and confirmed the official Stremio client has **no built-in SOCKS5/HTTP proxy support** at all. Workarounds like Proxifier exist for forcing a Windows desktop app through a proxy, but these are per-device hacks that don't extend to a TV app.
* Identified the approach people actually use to share one IP across multiple people/devices for Real-Debrid: proxy the *debrid stream URL itself* via an addon with built-in debrid-proxy support (e.g. Comet, MediaFlow Proxy), since that works identically regardless of which device or app is playing the stream.
* Re-examined the actual requirement and realized the real scope is narrow: only two specific devices need this (brother's Google TV and personal iPhone), not an open-ended group of friends.
* This narrower scope made Tailscale's built-in **exit node** feature viable after all — exit nodes are natively supported on Android TV (which Google TV runs) and iOS, no workarounds needed.
* Trade-off noted: an exit node routes *all* of a device's traffic, not just Stremio's. Acceptable for a TV (likely only running streaming apps anyway); on the iPhone this means toggling the exit node on/off manually around Stremio use rather than leaving it on permanently, since Android supports per-app split tunneling but iOS does not.
* Known limitation carried over: Real-Debrid may track concurrent stream count per account independent of IP, so this doesn't fully guarantee no restrictions if both people stream at the same time.
* **Decision:** use the Tailscale exit node approach for this two-device use case instead of Dante/MediaFlow Proxy. Dante remains installed and working in case a broader, multi-friend shared-proxy setup is wanted later.
